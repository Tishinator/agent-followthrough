Requirements - Revision 2

1. Functional Requirements

IDRequirement
FR-1The system shall provide a concrete registration interface that an OpenClaw agent can invoke when delegating or starting background work.
FR-2The system shall persist tracked task state in a durable local store independent of conversational memory.
FR-3The system shall use a scheduler independent of agent context to evaluate active tasks at a configured interval.
FR-4The system shall represent task states using explicit definitions: running, completed, failed, stale, unknown.
FR-5The system shall emit proactive updates for completion, failure, and overdue-running states.
FR-6The system shall suppress duplicate notifications using explicit last-notification metadata.
FR-7The system shall record evidence of observed task state, emitted updates, and scheduler activity.
FR-8The system shall support configurable check-in intervals, with a target default of 2–3 minutes when feasible.
FR-9The system shall survive agent/session restart without losing tracked active tasks.
FR-10The system shall support Phase 1 task types only where observation is concretely defined; unsupported task types shall be marked unsupported rather than silently treated as tracked.

2. Non-Functional Requirements

IDRequirement
NFR-1The system shall not depend on a human manually requesting status.
NFR-2The system shall distinguish observed, inferred, and unverified states in its outputs.
NFR-3The system shall emit honest degraded-status updates instead of false positives.
NFR-4The system shall be operable on the current Ubuntu/OpenClaw VM without major external infrastructure.
NFR-5The system shall minimize routine token/compute overhead by keeping Phase 1 check logic deterministic and non-LLM.

3. Interface Requirement

Phase 1 registration interface: a local command or script with explicit parameters (task id/title/owner/interval/observable handle/target). This interface must be specified before implementation begins.
