================================================================================
WHAT REV3 CLOSES
================================================================================

Emitter gate
  CLOSED. Architecture §4 now names the concrete command:
  openclaw message send --channel telegram --target <telegram-id> --message "..."
  Failure handling is also specified: failed emission is recorded as unverified
  delivery in task_events, not silently dropped.

Scheduler choice
  CLOSED. OpenClaw native cron explicitly selected with rationale documented
  (persistent, gateway-integrated, ecosystem-aligned). External crontab/systemd
  no longer mentioned.

Registration mechanism
  CLOSED. agent-follow-up is now defined as a local Node.js CLI in the project
  directory, invoked via local command execution. Command set documented:
  register, resolve, list. The registration contract in requirements §3 shows
  a concrete example invocation with all parameters.

Conditional observability
  CLOSED. "Conditional" has been removed from the Phase 1 observation table.
  Phase 1 now supports exactly two task classes:
    - Process-backed with owned marker/sentinel path (Yes)
    - Manual/marker-backed (Yes)
    - OpenClaw delegated session without explicit marker (No — Phase 2)
  Clean and unambiguous.

Unknown threshold N
  CLOSED. Default of 2 watchdog cycles is set in both FR-10 and architecture §6.
  The V&V proof expectation table also requires two consecutive failed observations
  before an unknown transition, confirming the implementation intent.

resolution_status vs last_observed_status
  CLOSED. Design §2 now explicitly states:
    last_observed_status = latest non-terminal observed state from the watchdog
    resolution_status    = confirmed terminal outcome used to close the task
  The distinction is clear and unambiguous.

Raw PID as observable_id
  CLOSED. Design §3 explicitly states Phase 1 shall not use a raw PID as
  primary observable identity. The preferred pattern is:
    observable_type = marker
    observable_id   = /absolute/path/to/task-marker.json
  PID is demoted to optional secondary evidence only.

================================================================================
REMAINING NOTES FOR IMPLEMENTATION
================================================================================

These are not blockers. They are small details the implementation will need to
resolve before or during coding.

1. MARKER FILE SCHEMA NOT DEFINED
   observable_id points to a JSON marker file but no document specifies what
   that file contains. The watchdog must parse it to determine running vs
   completed vs failed. A minimal schema needs to be defined before writing
   the watchdog evaluator. Suggested:
     {
       "status": "running" | "completed" | "failed",
       "exitCode": <int|null>,
       "updatedAt": "<ISO8601>",
       "message": "<optional string>"
     }

2. CRON JOB PAYLOAD TYPE NEEDS CLARIFICATION
   Architecture §1 says "OpenClaw cron job (isolated session or internal turn)."
   An isolated session uses an agentTurn payload, which invokes an LLM —
   contradicting NFR-5 (deterministic, non-LLM). The cron job should exec the
   agent-follow-up watchdog script directly, not drive an LLM turn. The cron
   job definition (a listed Phase 1 deliverable) must confirm a non-LLM
   execution path.

3. DATABASE PATH NOT SPECIFIED
   The design requires a dedicated operational state path outside of memory/
   but does not name one. A path must be chosen before implementation.
   Suggested: ~/projects/agent-follow-up/data/tasks.db

4. agent-follow-up resolve PARAMETERS NOT SPECIFIED
   The register command has a full parameter example. The resolve command is
   listed but its parameters are not documented. Likely shape:
     agent-follow-up resolve --id <id> --status completed|failed [--message "..."]
   Should be defined before the CLI is built.

5. "WRAPPER INTEGRATION" IS VAGUE (minor)
   Architecture §3 says agent-follow-up is "invoked via local command execution
   or wrapper integration." The latter is not defined. If it is not needed for
   Phase 1 it should be removed to avoid confusion during implementation.

6. FR-7 ID FORMATTING (cosmetic, persistent across all revisions)
   The requirements PDF table renders FR-7's ID in a broken layout due to the
   table column rendering. The requirement text itself is correct. No functional
   impact.
