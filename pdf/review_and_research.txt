------------

Runtime:       Node.js >= 22.12

OpenClaw is a personal AI assistant gateway. It routes messages between external
chat channels (Telegram, Slack, Discord, Signal, WhatsApp, etc.) and AI agents.
It runs as a persistent local WebSocket service (the Gateway) on port 18789.


CLI INTERFACE (40+ commands)
-----------------------------
openclaw gateway   run / start / stop / status / install / restart
openclaw agent     run a single agent turn, optionally deliver result to channel
openclaw sessions  list stored sessions (--active, --agent, --json)
openclaw cron      manage background jobs (add, list, run, edit, disable, enable, rm)
openclaw message   send/read messages (send, read, react, broadcast)
openclaw memory    search/reindex semantic memory
openclaw hooks     manage event hooks (list, enable, disable, install)
openclaw skills    list/inspect available skills
openclaw status    channel health and session summary
openclaw doctor    health checks and quick fixes
openclaw config    get/set/unset/validate config values


BUILT-IN CRON SYSTEM (key finding — adopted in Rev3)
------------------------------------------------------
OpenClaw has a production-grade internal cron scheduler. Rev3 correctly selects
this over external crontab/systemd.

Storage:    ~/.openclaw/cron/jobs.json (persists across restarts)
Run logs:   ~/.openclaw/cron/runs/<job-id>.jsonl (per-job execution history)
Execution:  Inside Gateway process, independent of all agent conversational context

Schedule types:
  - "at"    — one-shot timestamp
  - "every" — fixed interval (milliseconds)
  - "cron"  — 5-field cron expression with optional IANA timezone

Session targets:
  - "main"     — enqueue system event, runs on next heartbeat with main context
  - "isolated" — dedicated agent turn in a cron:<jobId> session

Delivery modes:
  - "announce" — posts result to a channel (Telegram, Slack, Discord, etc.)
  - "webhook"  — HTTP POST to a URL
  - "none"     — internal only

Key CLI:
  openclaw cron add --name "..." --every 3m --session isolated --message "..."
  openclaw cron add --name "..." --cron "*/3 * * * *" --tz America/New_York ...
  openclaw cron list
  openclaw cron run <job-id>
  openclaw cron runs --id <job-id>
  openclaw cron edit/disable/enable/rm <job-id>


SESSION SYSTEM
--------------
Sessions are stored as JSONL files per conversation.
Index: ~/.openclaw/agents/<agent-id>/sessions/sessions.json

Fields in sessions index:
  sessionId       — UUID (this is the correct format for owner_session_uuid)
  chatType        — "direct" or "group"
  deliveryContext — { channel, to }
  lastChannel     — last delivery channel used
  updatedAt       — timestamp

IMPORTANT: sessions --json returns metadata only. There is NO status field
indicating whether a session is currently executing or has completed its work.
A session exists once created and has no "done" state in the index. This is
why Rev3 defers delegated-session observation to Phase 2 — the correct call.

HEARTBEAT SYSTEM
----------------
Periodic lightweight agent check-in (default 30 min, configurable).
Uses lightContext (HEARTBEAT.md only) to minimize token use.
Current heartbeat model: ollama-remote/qwen3.5:9b
Not sufficient alone as a task watchdog — supplements but does not replace a
dedicated cron-based watchdog.


HOOKS SYSTEM

Relevant for Phase 2: a hook on agent lifecycle events could write completion

PART 4: REMAINING NOTES FOR IMPLEMENTATION
================================================================================

These are not blockers. They are details the implementation will need to resolve
and that are worth calling out before coding begins.

1. MARKER FILE SCHEMA NOT DEFINED
   observable_id points to a JSON file (e.g. /path/to/task-marker.json) but no
   document specifies what that file contains. The watchdog must parse it to
   determine running vs completed vs failed. A minimal schema is needed, e.g.:
     { "status": "running" | "completed" | "failed", "exitCode": <int|null>,
       "updatedAt": "<ISO8601>", "message": "<optional>" }
   This should be defined in the design doc before implementation of the
   watchdog evaluator.

2. CRON JOB PAYLOAD TYPE NEEDS CLARIFICATION
   Architecture §1 says "OpenClaw cron job (isolated session or internal turn)."
   If the payload kind is "agentTurn" (isolated session), an LLM is invoked,
   which contradicts NFR-5 (non-LLM, deterministic). The correct Phase 1
   approach is for the cron job to exec the agent-follow-up watchdog script
   directly rather than drive an LLM turn. The cron job definition (one of the
   Phase 1 deliverables in SPP §5) must specify a non-LLM execution path.

3. DATABASE PATH NOT SPECIFIED
   Design and architecture both require a "dedicated operational state path"
   separate from memory/ but no actual path is given. The implementation
   must choose one. Suggested: ~/projects/agent-follow-up/data/tasks.db
   or ~/.openclaw/agent-follow-up/tasks.db. Either works; just needs to be
   decided and documented.

4. "WRAPPER INTEGRATION" INVOCATION PATH IS VAGUE
   Architecture §3 says agent-follow-up is "invoked via local command execution
   or wrapper integration." Local command execution is clear. "Wrapper integration"
   is not defined. If this means an OpenClaw skill wrapper, that path should be
   documented. If it's not needed for Phase 1, the phrase should be removed to
   avoid confusion.

5. agent-follow-up resolve PARAMETERS NOT SPECIFIED
   Architecture lists register, resolve, and list as the command set but only
   register's parameters are shown in the requirements registration contract.
   The resolve command needs parameter documentation before the CLI is built.
   Likely: agent-follow-up resolve --id <id> --status completed|failed
           [--message "optional note"]

6. FR-7 ID FORMATTING (COSMETIC, PERSISTENT ACROSS ALL REVISIONS)
   In all three revisions the requirements PDF table renders FR-7's ID as
   "F R -" with "7" on the next line. The requirement content is correct; this
   is a PDF generation artifact. No functional impact.

PART 5: OVERALL FEASIBILITY — FINAL VERDICT
================================================================================

The project is achievable. Rev3 is implementation-ready.

The three-revision cycle correctly identified and closed every architectural risk:
- The core technical risk (OpenClaw session state not queryable) is handled by
  deferring to Phase 2 and requiring explicit markers as a prerequisite.
- The concurrency risk is handled by SQLite.
- The token overhead risk is handled by the non-LLM deterministic design constraint.
- The notification path is concrete and verified against the actual environment.
- The scheduler is now the correct choice for this ecosystem.

What is required before writing the first line of code:
  1. Define the marker file JSON schema (see note 1 above).
  2. Confirm the cron job payload type is exec/script, not agentTurn (note 2).
  3. Decide on the database file path (note 3).

What is ready to build (in milestone order per SPP §5):
  Milestone 1: agent-follow-up Node.js CLI (register, resolve, list)
               + SQLite schema + migration bootstrap
  Milestone 2: Deterministic watchdog + OpenClaw cron job definition
  Milestone 3: Notification emitter wrapper (openclaw message send)
  Milestone 4: V&V proof package execution

Phase 2 prerequisites (before delegated OpenClaw sessions can be tracked):
  - Define and document the completion marker convention that delegated agents
    must write to their output or a designated file.
  - Implement JSONL session log parser or marker-file hook in agent workflow.
