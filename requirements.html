Requirements

1. Functional Requirements

IDRequirement
FR-1The system shall allow an OpenClaw agent to register a background task with a durable identifier, description, owner session, start time, check-in interval, and target chat/session.
FR-2The system shall persist tracked task state in a durable local file so tracking survives agent/session restarts.
FR-3The system shall periodically evaluate active tasks using a scheduler independent of agent memory.
FR-4The system shall detect at minimum these states when observable: running, completed, failed, stale/overdue, and unknown.
FR-5The system shall send a proactive update when a task completes.
FR-6The system shall send a proactive update when a task fails or becomes blocked/overdue.
FR-7The system shall avoid duplicate notifications by recording the last sent update timestamp and status.
FR-8The system shall support a configurable check-in cadence, with an initial target of 2–3 minutes when feasible.
FR-9The system shall record evidence of updates sent and task state transitions.
FR-10The system shall work with OpenClaw background/delegated tasks, including spawned sessions or long-running local processes, to the extent that observable identifiers are available.

2. Non-Functional Requirements

IDRequirement
NFR-1The system shall not depend on a human operator manually requesting status.
NFR-2The system shall be transparent about what is observed vs inferred vs unverified.
NFR-3The system shall fail safely: if a task cannot be observed, it should emit an honest degraded-status update rather than false success.
NFR-4The system shall be simple enough to operate on the current Ubuntu VM running OpenClaw.
NFR-5The system shall preserve a low enough operational overhead to avoid materially increasing token or compute waste.

3. Constraints

  • Must integrate with OpenClaw as it exists today.
  • Must not rely on fake reminders or unenforced agent promises.
  • Must produce real, inspectable evidence.
