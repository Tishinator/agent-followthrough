Requirements - Revision 3

1. Functional Requirements

IDRequirement
FR-1The system shall provide a local Node.js CLI called agent-follow-up as the Phase 1 registration interface.
FR-2The system shall store task state in SQLite under a dedicated operational state path, independent of OpenClaw memory artifacts.
FR-3The system shall use OpenClaw native cron as the Phase 1 scheduler.
FR-4The system shall support these Phase 1 task classes only: process-backed tasks with owned marker files, and manual/marker-backed tasks.
FR-5The system shall use a deterministic watchdog and shall not require an LLM for routine reminder decisions.
FR-6The system shall emit notifications using a concrete OpenClaw-native message command.
FR-7The system shall suppress duplicate notifications using explicit notification metadata.
FR-8The system shall use a configurable check-in interval, with a target default of 2–3 minutes when feasible.
FR-9The system shall classify tasks using explicit state definitions: running, completed, failed, stale, unknown.
FR-10The system shall transition a task to unknown after the configured unknown threshold; the default Phase 1 threshold shall be 2 watchdog cycles.

2. Non-Functional Requirements

IDRequirement
NFR-1The system shall not depend on a human manually asking for updates.
NFR-2The system shall distinguish observed, inferred, and unverified states.
NFR-3The system shall emit honest degraded-state updates rather than false success.
NFR-4The system shall operate within the existing Ubuntu/OpenClaw environment.
NFR-5The system shall minimize token/compute overhead by keeping Phase 1 evaluation deterministic and non-LLM.

3. Registration Contract

Phase 1 contract direction:

agent-follow-up register --id <id> --title <title> --interval 3 --target telegram:8625301893 --observable-type marker --observable-id /path/to/marker.json
